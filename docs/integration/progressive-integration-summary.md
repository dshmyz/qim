## 🔍 应用逻辑集成分析

### 发现的问题

**应用逻辑过于复杂**：

1. **openApp 函数**（120+ 行）
   - 处理内置应用 code 映射
   - 从应用分类查找应用信息
   - 记录最近使用的应用
   - 处理特殊应用（短链接、小程序）
   - 根据 openType 决定打开方式（内部/外部）
   - Electron 环境和浏览器环境的兼容处理

2. **相关状态和函数**：
   - `recentApps` - 最近使用的应用
   - `activeAppTab` - 应用面板的tab切换
   - `currentUserApp` - 当前打开的用户应用
   - `showMiniAppList` - 小程序面板状态
   - `addToRecentApps` - 添加到最近使用的应用
   - `resolveAppId` - 解析应用ID
   - `openApp` - 打开应用
   - `openUserApp` - 打开用户创建的应用
   - `handleSwitchApp` - 处理切换应用

3. **依赖关系复杂**：
   - 依赖 `appCategories`
   - 依赖 `systemConfigStore`
   - 依赖 `selectedAppId`（已在 useAppState 中）
   - 依赖 Electron API
   - 依赖 localStorage

### 风险评估

| 风险项 | 等级 | 说明 |
|--------|------|------|
| 代码复杂度 | 高 | 120+ 行的 openApp 函数 |
| 依赖关系 | 高 | 依赖多个外部状态和模块 |
| 特殊情况处理 | 高 | 多种打开方式和环境兼容 |
| 测试难度 | 高 | 需要测试多种场景 |
| 回滚难度 | 中 | 如果出问题，回滚较容易 |

**综合风险等级**: 🔴 高

### 建议

**强烈建议**: 暂时跳过应用逻辑集成

**理由**:
1. 风险过高，不符合渐进式集成的原则
2. 收益相对有限（应用逻辑相对独立）
3. 可能引入难以发现的 bug
