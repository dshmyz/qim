# 功能完善实施方案

## 1. 权限问题 - 发布系统消息权限

### 1.1 功能描述
实现系统消息发布权限控制，确保只有授权用户能够发布系统消息，避免滥用系统消息功能。

### 1.2 技术实现方案

#### 1.2.1 数据库结构修改
1. **新增 `UserRole` 表**
   ```go
   type UserRole struct {
       ID        uint      `json:"id" gorm:"primarykey"`
       UserID    uint      `json:"user_id" gorm:"not null;index;uniqueIndex:idx_user_role"`
       Role      string    `json:"role" gorm:"size:50;not null;uniqueIndex:idx_user_role"` // system_admin, system_publisher, etc.
       CreatedAt time.Time `json:"created_at"`
       User      User      `json:"user,omitempty" gorm:"foreignkey:UserID"`
   }
   ```

2. **修改 `SystemMessage` 表**
   - 增加 `Status` 字段，支持消息审核流程
   - 增加 `TargetType` 和 `TargetID` 字段，支持定向发送

#### 1.2.2 后端实现
1. **新增权限验证中间件**
   ```go
   func RequireRole(roles ...string) gin.HandlerFunc {
       return func(c *gin.Context) {
           userID, _ := c.Get("user_id")
           var userRoles []model.UserRole
           if err := database.GetDB().Where("user_id = ?", userID).Find(&userRoles).Error; err != nil {
               c.JSON(http.StatusForbidden, gin.H{"code": 403, "message": "无权限操作"})
               c.Abort()
               return
           }
           
           hasRole := false
           for _, ur := range userRoles {
               for _, role := range roles {
                   if ur.Role == role {
                       hasRole = true
                       break
                   }
               }
               if hasRole {
                   break
               }
           }
           
           if !hasRole {
               c.JSON(http.StatusForbidden, gin.H{"code": 403, "message": "无权限操作"})
               c.Abort()
               return
           }
           
           c.Next()
       }
   }
   ```

2. **系统消息相关API**
   - `POST /api/system-messages` - 发布系统消息（需要 `system_publisher` 或 `system_admin` 角色）
   - `GET /api/system-messages` - 获取系统消息列表
   - `PUT /api/system-messages/:id` - 更新系统消息状态（需要 `system_admin` 角色）

#### 1.2.3 前端实现
1. **新增系统消息管理组件**
   - 仅对有权限的用户显示
   - 支持创建、编辑、查看系统消息
   - 支持消息定向发送

2. **权限检查逻辑**
   ```typescript
   // 检查用户是否有系统消息发布权限
   const hasSystemMessagePermission = computed(() => {
     return user.value?.roles?.includes('system_publisher') || user.value?.roles?.includes('system_admin');
   });
   ```

### 1.3 测试方案
1. **权限验证测试**
   - 无权限用户尝试发布系统消息，应返回403错误
   - 有权限用户尝试发布系统消息，应成功
   - 普通用户查看系统消息，应成功

2. **消息发布测试**
   - 测试发布普通系统消息
   - 测试发布定向系统消息
   - 测试消息审核流程

## 3. 讨论组和群组的区别

### 3.1 功能描述
明确区分讨论组和群组的功能差异，优化用户体验。

### 3.2 技术实现方案

#### 3.2.1 前端实现
1. **会话列表区分**
   - 为讨论组和群组显示不同的图标
   - 在会话名称旁显示类型标识

2. **群聊设置界面**
   - 群组：显示群主、管理员设置，支持转让群主、设置管理员
   - 讨论组：显示创建者信息，不支持权限管理

3. **成员管理**
   - 群组：群主和管理员可以移除成员
   - 讨论组：所有成员可以邀请新成员，但不能移除成员

#### 3.2.2 后端实现
1. **权限验证优化**
   - 群组：严格的角色权限控制
   - 讨论组：简化的权限控制，所有成员权限相同

2. **API优化**
   - 为讨论组和群组提供不同的管理API
   - 讨论组不支持管理员设置和成员移除操作

### 3.3 测试方案
1. **功能区分测试**
   - 测试群组的权限管理功能
   - 测试讨论组的平等参与特性
   - 测试两种类型会话的UI区分

2. **权限验证测试**
   - 测试群组中普通成员无法移除其他成员
   - 测试讨论组中成员可以邀请新成员
   - 测试讨论组中成员无法移除其他成员

## 4. 频道的作用

### 4.1 功能描述
实现频道功能，作为单向信息发布渠道，用于企业内部通知、部门公告等场景。

### 4.2 技术实现方案

#### 4.2.1 数据库结构
1. **新增 `Channel` 表**
   ```go
   type Channel struct {
       ID          uint           `json:"id" gorm:"primarykey"`
       Name        string         `json:"name" gorm:"size:200;not null"`
       Description string         `json:"description" gorm:"type:text"`
       Avatar      string         `json:"avatar" gorm:"size:500"`
       CreatorID   uint           `json:"creator_id" gorm:"not null"`
       Status      string         `json:"status" gorm:"size:20;default:'active'"`
       CreatedAt   time.Time      `json:"created_at"`
       UpdatedAt   time.Time      `json:"updated_at"`
       DeletedAt   gorm.DeletedAt `json:"-" gorm:"index"`
       Creator     User           `json:"creator,omitempty" gorm:"foreignkey:CreatorID"`
       Subscribers []ChannelSubscriber `json:"subscribers,omitempty" gorm:"foreignkey:ChannelID"`
   }
   ```

2. **新增 `ChannelSubscriber` 表**
   ```go
   type ChannelSubscriber struct {
       ID        uint      `json:"id" gorm:"primarykey"`
       ChannelID uint      `json:"channel_id" gorm:"not null;index;uniqueIndex:idx_channel_user"`
       UserID    uint      `json:"user_id" gorm:"not null;index;uniqueIndex:idx_channel_user"`
       JoinedAt  time.Time `json:"joined_at"`
       Channel   Channel   `json:"channel,omitempty" gorm:"foreignkey:ChannelID"`
       User      User      `json:"user,omitempty" gorm:"foreignkey:UserID"`
   }
   ```

3. **新增 `ChannelMessage` 表**
   ```go
   type ChannelMessage struct {
       ID        uint           `json:"id" gorm:"primarykey"`
       ChannelID uint           `json:"channel_id" gorm:"not null;index"`
       SenderID  uint           `json:"sender_id" gorm:"not null"`
       Content   string         `json:"content" gorm:"type:text;not null"`
       Type      string         `json:"type" gorm:"size:20;not null;default:'text'"`
       CreatedAt time.Time      `json:"created_at"`
       DeletedAt gorm.DeletedAt `json:"-" gorm:"index"`
       Channel   Channel        `json:"channel,omitempty" gorm:"foreignkey:ChannelID"`
       Sender    User           `json:"sender,omitempty" gorm:"foreignkey:SenderID"`
   }
   ```

#### 4.2.2 后端实现
1. **频道相关API**
   - `POST /api/channels` - 创建频道（需要 `system_admin` 角色）
   - `GET /api/channels` - 获取频道列表
   - `POST /api/channels/:id/subscribe` - 订阅频道
   - `POST /api/channels/:id/unsubscribe` - 取消订阅频道
   - `POST /api/channels/:id/messages` - 发布频道消息（需要频道管理员权限）
   - `GET /api/channels/:id/messages` - 获取频道消息

2. **权限控制**
   - 频道创建：需要 `system_admin` 角色
   - 消息发布：需要频道管理员权限
   - 订阅/取消订阅：所有用户可操作

#### 4.2.3 前端实现
1. **频道列表组件**
   - 显示所有可用频道
   - 显示订阅状态
   - 支持快速订阅/取消订阅

2. **频道详情组件**
   - 显示频道信息和描述
   - 显示频道消息列表
   - 支持消息发布（仅频道管理员）

3. **通知集成**
   - 新频道消息推送通知
   - 频道订阅状态变更通知

### 4.3 测试方案
1. **频道功能测试**
   - 测试创建频道
   - 测试订阅/取消订阅频道
   - 测试发布频道消息
   - 测试获取频道消息

2. **权限验证测试**
   - 测试普通用户无法创建频道
   - 测试普通用户无法发布频道消息
   - 测试所有用户可以订阅频道
